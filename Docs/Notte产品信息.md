# Notte 产品概览

## 1. 产品定位

Notte 可以定位为一款**强 Apple 生态导向、低门槛、轻学习成本、以内容组织与表达体验为核心的下一代笔记与知识组织应用**。

**Notte 是一款面向 Apple 生态的结构化知识工作台。**

它通过 **Collection / Page / Node** 的结构模型，把记录、整理、规划、复用和输出统一在同一个系统中，并保持接近 Apple Notes 的低学习成本。

目标不是做另一个复杂生产力平台，而是让用户能够：

- 快速记录
- 自然形成结构
- 逐步整理内容
- 长期沉淀知识

它不是简单复制 Apple Notes、Notion 或 Heptabase，而是试图在三类能力之间建立自己的组合优势：

1. **像 Apple Notes 一样自然、轻量、几乎零学习门槛**；
2. **像模块化知识工具一样具备更强的结构化组织能力**；
3. **像面向创作者与长期知识积累的产品一样，强调内容块、模板、规划、跨页面复用与生态扩展**。

从目前的设想看，Notte 更适合被理解为：

> **“面向 Apple 生态用户的轻量级结构化知识工作台。”**

## 2. 核心差异

Notte 与现有产品的最大区别是：

> **把“文档”变成“模块化内容系统”**

在 Notte 中：

- 内容以 **Node** 为组织单位，Node 内容区承载具体内容
- Node 可以跨 Page 复用
- Page 可以随时重构
- Collection 可以规划整个知识体系

因此 Notte 不只是写文档，而是：

> **搭建内容结构**

---

## 3. 核心设计模型

Notte 的所有功能都建立在一个简单且统一的数据模型之上。

```
Collection
   ├ Collection（子 Collection，MVP 阶段不支持）
   ├ Page
   │   ├ Node
   │   │   ├ Node（可嵌套，深度不限）
   │   │   └ Node
   │   └ Node
   └ Page
```

三个核心对象：

### 3.1 Collection

Collection 是 **内容组织单元**。

用途：

- 项目空间
- 课程体系
- 知识专题
- 研究课题
- 内容仓库

Collection 可以包含：

- Page
- 子 Collection（用于更深层的主题划分，MVP 阶段不支持）

示例：

```
Swift课程 (Collection)
   ├ 课程规划 (Page)
   ├ 编程核心概念 (Collection)
   ├ SwiftUI (Collection)
```

---

### 3.2 Page

Page 是 **完整内容页面**。

类似：

- 一篇文章
- 一节课程
- 一份研究记录

Page 由多个 Node 组成。

示例：

```
第一节课 (Page)
   ├ Swift介绍 (Node)
   ├ Swift特点 (Node)
   └ Swift应用 (Node)
```

---

### 3.3 Node

Node 是 **内容模块**，也是 Notte 的核心单位。

Node 采用**大纲模型**：每个 Node 是一个可嵌套的大纲条目，拥有标题和内容区两个部分。

**Node 的结构：**

- **标题**：每个 Node 必有标题，是大纲视图中显示的那一行文字
- **内容区**：Node 标题下方的内容，可以包含文字、图片、代码等不同类型的内容块
- **子 Node**：Node 可以无限嵌套，形成树状大纲结构

**Node 的层级即缩进**，缩进深度同时决定标题的视觉级别（对应 Markdown 的 h1-h6），没有额外的类型之分。

**Node 内容区支持的内容类型（按阶段扩展）：**

- 文字段落（MVP）
- 列表项（POST）
- 图片（POST）
- 代码块（POST）
- 引用（POST）

Node 可以：

- 在 Page 中自由移动和重排
- 折叠 / 展开，收起整个子树
- 在不同 Page 之间迁移
- 被引用或复用（POST）

## 4. 视图系统

同一结构可以用不同视图呈现。

### 4.1 Outline View（大纲视图）

适合编辑和写作。

```
Node
   Node
   Node
```

用户可以：

- 写内容
- 折叠结构
- 调整顺序

---

### 4.2 Map View（思维导图视图）

适合规划和全局浏览。

```
      Node
     /    \
  Node   Node
```

作用：

- 课程结构设计
- 文章提纲规划
- 专题知识整理

## 5. 核心功能

### 5.1 模块化编辑与跨页面移动

Notte 的差异化之一，是把内容当作“模块”而不是“整页静态文本”。

核心能力包括：
- Node 在 Page 中自由重排；
- Node 在不同 Page 之间迁移；
- 后续支持 Node 复制、引用、嵌入、模板化；
- 将“写作”与“搭建内容结构”统一起来。

这会让 Notte 更适合：
- 课程内容整理；
- 研究笔记拆解；
- 长文写作；
- 项目文档搭建；
- 复用型知识库建设。

### 5.2 模板体系

提出的 Node 模板库，是一个很重要的增长点。

模板能力可以分为三层：

#### Node 模板
- 标准笔记卡片；
- 阅读摘要卡；
- 会议记录块；
- 任务块；
- 课程知识点块；
- 代码说明块；
- 研究结论块。

#### Page 模板
- 周报页；
- 学习页；
- 项目页；
- 读书笔记页；
- 课程讲义页；
- 产品规划页。

#### Collection 模板
- 知识专栏；
- 学习路径；
- 课程体系；
- 研究课题；
- 项目管理空间；
- 创作者内容库。

模板库如果支持上传、下载、分享，会带来三个直接收益：
- 降低新用户上手门槛；
- 增强社区传播；
- 让产品从工具变成“方法 + 内容结构”的平台。

### 5.3 自动思维导图转换

`Page / Node`可以无缝转换为思维导图进行编辑，在同一窗口内；并在思维导图保留大纲模式的易用性，提供更直观、全局化的显示模式，便于展示、规划。

### 5.4 规划模式

设想的“规划功能”，本质上不是普通大纲，而是**面向 Page / Collection 的内容结构设计能力**。

可分为两种层级：

#### Collection 级规划
- 最适合作为主规划场景；
- 用于设计这个专题下应该有哪些 Page；
- 每个 Page 放哪些 Node；
- 每部分承担什么主题；
- 各内容块之间如何关联。

#### Page 级轻规划
- 更轻量；
- 可以是节点布局建议；
- 可以是标签化组织；
- 可以与思维导图或结构图结合。

如果最终采用“思维导图式规划”，会更贴近：
- 学习路线设计；
- 课程结构规划；
- 专题拆解；
- 写作提纲到正文的转换。

### 5.5 搜索

可在`Collection / Page`中**以Node为单位**搜索，可选择搜索 **Node 标题** 或 **Node 内容区文字**。

同时提供`Page`内文本搜索，方便快速定位

### 5.6 协作与共享

已经提出：协作不一定只发生在整篇文档内部，而可以发生在不同粒度上。

Notte 可考虑三种协作单元：
- **Page 级协作**：共同编辑一页内容；
- **Node 级协作**：围绕某个模块进行修改、评论、补充；
- **Collection 级共享**：把一个完整主题空间共享给别人。

其中，Collection 级共享很有潜力，因为它更像“专题空间分享”，天然适合：
- 课程；
- 研究项目；
- 知识专栏；
- 团队资料库；
- 模板包分发。

### 5.7 导入、导出与内容整理能力

从此前的设想看，Notte 可以与另一个专注于文件整理与排版的工具形成协同。

因此，Notte 在导入导出上应重点考虑：
- Markdown 导入导出；
- PDF、图片格式的大纲导出
- 富文本 / PDF / 图片 / 附件组织；
- 面向排版与输出的文档整理；
- 与外部文件结构的映射；
- 面向发布、归档、复用的内容输出。
- 直接导出思维导图为图片、PDF，而非大纲

这部分不一定要做“AI 生成”，但应把“内容整理效率”做到非常顺手。

### 5.8 本地 MD 文件联动编辑

**阶段：POST-MVP（优先级较高，建议作为首个破圈功能）**

Notte 支持打开本地 `.md` 文件，并将其关联为某个 Collection 下的一个 Page。

**核心工作流：**

```
本地 .md 文件（iCloud Drive / 本地存储）
    ↓ 用户通过 Files App 或 Share Sheet 导入
Notte 解析 MD 结构 → 生成 Page
    Heading 层级 → Node depth（h1 = depth 0，h2 = depth 1，以此类推）
    段落内容 → Block（type: text）
    ↓ 用户在 Notte 中编辑 Page
保存时自动写回原始 .md 文件
    （通过 Security Scoped Bookmark 持久持有文件访问权限）
```

**数据模型变更（Page 新增两个字段）：**

```
linkedFilePath:     String?   // 显示用，文件名路径提示
linkedFileBookmark: Data?     // Security Scoped Bookmark，持久访问文件
```

`linkedFileBookmark != nil` 时，该 Page 为"文件关联模式"，保存时自动触发写回。

**映射规则（可逆）：**

| Markdown | Notte |
|---|---|
| `# 标题` | depth=0 Node |
| `## 二级标题` | depth=1 Node |
| 普通段落 | 上方最近标题 Node 的 Block |
| 不支持的语法（表格、footnote 等） | 以 rawMarkdown 类型 Block 原样保留，写回时原样输出 |

**用户场景：**

开发者 / 技术写作者可以：
1. 在 Mac 上用 Xcode / VS Code 维护项目文档（.md）；
2. 出门在外用 iPhone 打开 Notte，找到对应 MD 文件关联的 Page；
3. 在结构化大纲编辑器里修改内容；
4. 保存后文件自动同步回 iCloud Drive / Git 仓库。

**核心价值：**
- 保留 `.md` 文件的非二进制、可 Git 追踪特性；
- 同时获得 Notte 结构化大纲编辑器的移动端体验；
- 让 Notte 成为开发者和技术写作者在 iPhone 上编辑项目文档的最佳移动端入口。

**注意事项：**
- 外部修改检测：打开 Page 时做 hash 校验，如文件已被外部修改，提示用户选择是否重新导入；
- UI 需有明确的"文件关联"标识（文件图标 + 文件名），让用户清楚当前 Page 的改动会写回磁盘。

### 5.9 Apple 生态深度整合

这是 Notte 最值得押注的方向之一。

潜在核心点包括：
- iPhone / iPad / Mac 一体化体验；
- iCloud 同步；
- Share Sheet 快速收藏；
- 快捷指令集成；
- Apple Pencil 场景；
- 深色模式、系统字体、原生手势、拖放、Spotlight、文件系统协同；
- 更贴合 Apple 用户的交互与审美。

如果这条路线走得足够彻底，Notte 就不只是“一个笔记 App”，而是“Apple 用户更愿意长期驻留的知识空间”。

## 6.存储架构

> **local-first + iCloud 轻同步 + 大文件分离。**

#### 6.1 核心设计
- **永远提供离线使用**：可以在本地搭建`Collections`，完全不依赖云服务、云存储
- **离线使用为核心**：同步只是附加
- **可打开本地`Colletion / Page`文件**：文件默认存`CoreData`，可**指定另存为**为文件

#### 6.2 iCloud同步

**额外**提供跨设备iCloud同步，并**不是**核心存储方案

- **同步核心结构化数据放在 iCloud**：包括 `Node / Page / Collection`、正文文本、链接关系、页面组织结构等；
- **索引、缓存、本地预览留在本地**：搜索索引、缓存文件、预览数据不占用云端主存储；
- **附件默认本地存储**：图片、PDF、音视频、导出文件等默认保存在本地；
- **附件云同步由用户选择开启**：用户在有需要时再决定是否把附件同步到云端；
- **大体积导出文件走 Files 而不是 iCloud 主数据区**：避免导出内容无节制占用同步空间。

#### 6.3 对于未订阅iCloud的用户

- **纯本地服务**：所有内容存放在本地，使用 SwiftData
- **后期开发Notte Cloud**：提供协作、跨平台服务

#### 6.4 这套方案的意义
- 能让**未订阅大容量 iCloud 的用户**也正常使用 Notte 的核心能力；
- 保持 Apple 生态下的自然同步体验，同时避免一开始就被 5GB 免费空间卡死；
- 降低在产品初期自建 Notte Cloud 的资金压力；
- 给未来扩展留出空间：后续若要支持协作、跨平台、团队能力，再逐步补充 Notte Cloud 层。

#### 6.5 对产品定位的影响
这意味着 Notte 的存储设计不是“纯 iCloud 依赖”，也不是“一开始就重后端平台化”，而是：

> **先用 local-first 保障可用性，用 iCloud 承载轻量核心同步，再把真正高成本的协作与跨平台能力留给未来的云层扩展。**

这套策略和 Notte 当前“强 Apple 生态、低门槛、低成本起步”的路线是高度一致的。

## 7. 价值追求

### 7.1 把复杂知识工具变得真正可用

很多知识管理产品功能很强，但学习成本高、概念重、日常记录阻力大。Notte 的价值之一，是把这类工具的优点重新包装成“自然可用”的体验。

目标不是让用户先理解复杂方法论，而是：
- 先开始写；
- 写着写着自然形成结构；
- 在需要时再逐步进入规划、模板、组织和复用。

### 7.2 用更低门槛支持更长期的知识沉淀

大量用户不是没有记录需求，而是无法持续。原因通常不在“不会记”，而在：
- 工具太重；
- 组织成本太高；
- 后续整理困难；
- 之前写的内容难复用。

Notte 的价值，是把“记录”与“后续组织”之间的断裂缩小，让内容从一开始就是可迁移、可拼装、可归类的。

### 7.3 在美观、结构化、轻便之间取得平衡

市场上很多产品只能满足其中一部分：
- 有的轻便但弱结构；
- 有的强结构但太复杂；
- 有的设计好看但不适合长期知识管理。

Notte 的理想形态，是在以下三者之间建立平衡：
- **足够轻**：快速记；
- **足够美**：愿意长期用；
- **足够结构化**：可沉淀、可输出、可协作。

### 7.4 从“笔记工具”走向“内容工作空间”

如果 Page / Node / Collection、模板、规划、共享这些能力成型，Notte 的定位会逐步从笔记 App 升级为：
- 个人知识空间；
- 学习与课程工作台；
- 创作者内容生产空间；
- 小团队轻协作空间。

## 8. 市场前景

### 8.1 市场仍然存在机会，但不是“通用笔记 App”机会

笔记与知识管理赛道并未结束，但“再做一个大而全笔记工具”的空间已经明显变窄。真正还有机会的方向通常是：
- 面向特定生态；
- 面向特定人群；
- 面向特定工作流；
- 在体验上形成明显差异。

Notte 的机会不在于正面复制 Notion，而在于：
- 对 Apple 生态做得更深；
- 对轻量结构化内容组织做得更自然；
- 对创作者、学习者、研究者的长期使用更友好；
- 用模板和规划能力形成自己的方法论壁垒。

### 8.2 Apple 生态是现实机会点

Apple Notes 已经覆盖了大量基础记录需求，并且原生支持附件、扫描、手写、协作和 iCloud 同步。Apple 官方资料显示，Notes 已支持共享协作、文件夹协作、扫描文档、附件管理以及跨设备同步等能力。

这意味着：
- 单纯做“更好记事本”很难打；
- 但如果做出**更强结构化 + 更强模板化 + 更强规划性 + 更强输出能力**，仍然有差异化空间。

换句话说，Apple 生态本身不是空白市场，但它是一个**高质量存量市场**：用户付费能力较好，审美要求较高，也愿意为顺滑体验买单。

### 8.3 知识工作正从“记录”转向“组织与再利用”

当前主流产品普遍在强化：
- AI 辅助；
- 知识连接；
- 协作；
- 结构化内容；
- 发布与分享。

例如：
- Notion 正在强化 AI workspace、知识库、文档与项目协同能力。citeturn243913search9turn243913search13turn243913search17
- Obsidian 继续强化本地优先、链接化知识组织、插件生态与发布能力。
- Heptabase 持续强调白板式视觉知识组织、双向链接、PDF 标注与复杂主题学习研究。
- Capacities 主打 object-based note-taking，把内容视为对象而不是传统文件。
- Craft 则强调更强的写作体验、模板、发布和更好的苹果端呈现。

这说明 Notte 所在方向并不是伪需求；相反，需求在升级。但竞争也已经进入“产品哲学 + 使用体验 + 生态能力”的较量阶段。

### 8.4 最有希望的切入方式是“小而尖”

Notte 更现实的前景，不是一下子覆盖所有人，而是先抓住某一小群用户：
- Apple 重度用户；
- 学习型用户；
- 课程创作者；
- 研究与阅读型用户；
- 希望兼顾美感与结构的知识工作者。

只要在这些人群中形成高留存，产品就有机会从“工具”变成“方法平台”。

## 9. 主要挑战

### 9.1 与大平台正面竞争非常难

| Apple Notes   |      免费、系统预装、同步自然；      |
| ------------- | :----------------------------------: |
| **Notion**    |       **用户基础大、品牌强；**       |
| **Obsidian**  |        **社区强、生态成熟；**        |
| **Heptabase** | **在视觉知识管理领域已有鲜明定位；** |

因此，Notte 不能靠“我也有这些功能”取胜，必须靠：

- 更明确的定位；
- 更顺滑的流程；
- 更清晰的人群价值；
- 更高频、更长期的真实场景。

### 9.2 产品边界容易失控

现在的设想很有潜力，但也天然有风险：
- 想做轻笔记；
- 又想做结构化知识管理；
- 又想做模板市场；
- 又想做规划；
- 又想做协作；
- 又想做导出与发布。

如果没有优先级，产品很容易变重，最后失去“低门槛”的初衷。

### 9.3 数据同步与存储策略会影响用户体验

目前已经确认的方向，是**local-first + iCloud 轻同步 + 大文件分离**。这比“全量依赖 iCloud”更现实，也更适合产品初期。

其优势在于：
- 更贴合 Apple 用户心智；
- 核心数据可同步，但不强迫所有数据都占用 iCloud；
- 免费 iCloud 用户也更容易持续使用；
- 初期后端压力较小，适合低成本起步。

但挑战仍然存在：
- 需要明确区分“核心数据、缓存、附件、导出文件”的边界；
- 同步冲突与跨设备一致性仍要处理好；
- 用户可能会关心“本地附件”和“云端核心数据”之间的可理解性；
- 如果后续要做团队协作、跨平台和 AI 服务，仅靠这套方案仍然不够，最终仍需要 Notte Cloud。

### 9.4 “看起来很好”不等于“用户会持续使用”

知识管理产品常见问题不是首次惊艳，而是长期留存差。核心在于：
- 用户能否迅速开始；
- 写进去的东西以后能否轻松找到；
- 是否真的提高了整理效率；
- 是否形成自己的内容资产。

Notte 必须回答的，不是“功能是否先进”，而是“用户为什么三个月后还留在这里”。

### 9.5 商业化需要足够克制

如果过早堆付费点，会伤害体验；
如果过晚考虑商业化，又可能难以持续投入。

比较合理的方向通常是：
- 免费基础使用；
- 高级模板 / 模板市场；
- 更强导出与发布；
- 协作功能；
- 更高容量与高级同步；
- 面向创作者或课程场景的增值能力。

## 10. 市场上的竞争者与同类应用

### 10.1 Apple Notes

**特点：**
- 系统级入口；
- 免费；
- 同步自然；
- 支持扫描、附件、录音转写、共享协作。

**对 Notte 的启示：**
- 基础记录体验必须极顺；
- Apple 生态整合必须足够深；
- 不能只靠“记笔记”与它竞争。

### 10.2 Notion

**特点：**
- 文档、数据库、协作、AI、项目管理一体化；
- 适合团队与较复杂工作流；
- 生态与品牌强。

**对 Notte 的启示：**
- 不宜走“大而全通用工作台”路线；
- 应坚持轻量、原生体验和特定场景深耕。

### 10.3 Obsidian

**特点：**

- 本地优先；
- Markdown；
- 双向链接；
- 插件生态非常强；
- 适合深度知识管理用户。

**对 Notte 的启示：**
- 高阶用户愿意为“知识结构”投入；
- 但大众用户未必愿意学习复杂系统；
- Notte 可走“更低学习成本”的路线；
- Obsidian 移动端体验是明显短板，Notte 的本地 MD 文件联动功能（5.8 节）直接针对这一痛点：开发者可以继续用 Obsidian / VS Code 维护 `.md` 文件，用 Notte 作为移动端的结构化编辑入口，两者形成互补而非替代关系。

### 10.4 Heptabase

**特点：**
- 白板式组织；
- 复杂主题学习与研究；
- 视觉化知识连接；
- PDF 标注、双链能力突出。

**对 Notte 的启示：**
- 视觉规划和知识拆解是强需求；
- 但 Heptabase 偏研究型，门槛相对更高；
- Notte 可把“规划”做得更轻、更生活化、更适合日常内容生产。

### 10.5 Craft

**特点：**
- 文档美感强；
- Apple 端体验好；
- 模板、发布、团队协作能力成熟。

**对 Notte 的启示：**
- 高审美和高完成度确实有市场；
- Apple 用户愿意为体验买单；
- Notte 若走苹果生态精品路线，Craft 是非常值得重点研究的对手。

### 10.6 Capacities

**特点：**
- Object-based note-taking；
- 强调把信息当作对象组织；
- 更适合长期知识积累与关系连接。

**对 Notte 的启示：**

- 的 Node / Page / Collection 体系，本质上也在靠“内容对象化”建立优势；
- 但 Notte 可以比 Capacities 更强调原生体验与低门槛。

## 11. 目标人群

### 11.1 第一目标人群：Apple 生态重度用户

这类用户通常：
- 长期使用 iPhone、iPad、Mac；
- 在乎原生体验、同步体验和界面美感；
- 不喜欢过重、过丑、过工程化的工具；
- 愿意为顺滑体验付费。

这是 Notte 最适合优先服务的一群人。

### 11.2 学习者与研究型用户

包括：
- 大学生；
- 研究生；
- 博士；
- 终身学习者；
- 课程学习者；
- 需要长期积累阅读、思考与输出的人。

他们需要：
- 既能快速记录；
- 又能逐步形成体系；
- 还能把碎片内容转化为结构化成果。

### 11.3 内容创作者与课程生产者

包括：
- 写作者；
- 教程作者；
- 博主；
- 课程开发者；
- 知识付费创作者；
- 需要反复复用内容模块的人。

这类人会特别需要：
- Node 复用；
- 模板；
- Collection 级内容规划；
- 输出与发布能力。

### 11.4 小团队与轻协作场景用户

包括：
- 两三人的小团队；
- 课程共创团队；
- 小型研究团队；
- 顾问 / 自由职业者协作。

他们通常不一定需要复杂项目管理，但需要：
- 共享页面；
- 共享专题空间；
- 协同维护内容资产。

### 11.5 不建议作为第一阶段主打的人群

以下人群可以服务，但不建议第一阶段强打：
- 大型企业团队；
- 高流程化项目管理团队；
- 纯 Markdown / 极客工作流重度用户；
- 只追求免费替代品的泛用户。

原因是这些人群要么需求过重，要么价格敏感，要么已有强替代工具。

## 12. 对 Notte 的总体判断

### 12.1 值得做，但必须明确“差异化战场”

Notte 有现实机会，但前提不是功能堆叠，而是明确回答下面这个问题：

> **为什么用户不继续留在 Apple Notes、Notion、Obsidian、Heptabase 或 Craft，而要选择 Notte？**

一个较清晰的答案可以是：

> **因为 Notte 在 Apple 生态里，把“轻记录、结构化组织、内容规划、模板复用、长期沉淀”这几件事做到了一个更自然、更美、更低门槛的平衡点。**

### 12.2 最好的打法是“先窄后宽”

建议先把第一阶段产品收敛为：
- 面向 Apple 用户；
- 面向学习 / 创作 / 研究；
- 以 Page / Node / Collection 为核心；
- 先把模板、结构化编辑、规划和导出体验做扎实；
- 不急于做大而全协作。

### 12.3 真正的护城河不会只是功能，而是使用感觉

最终决定 Notte 成败的，很可能不是某一个单点功能，而是用户在日常使用中形成的综合感受：
- 打开就想记；
- 记完很容易整理；
- 整理后很容易复用；
- 复用后能形成成果；
- 长期积累不会乱。

如果这条链路被打通，Notte 就有机会从“又一个笔记工具”变成“某类用户离不开的内容空间”。

---

## 13. 总结

> **Notte 最有潜力的方向，不是成为另一个全能型笔记平台，而是成为 Apple 生态下，兼顾轻量、结构、美感与长期知识沉淀的一体化内容工作空间。**

Notte 的核心并不是某一个功能，而是一个统一的结构模型：

```
Collection
   └ Page
       └ Node（可嵌套，标题 + 内容区）
```

在这个结构之上，Notte整合：

- 记录
- 规划
- 整理
- 复用
- 输出

最终形成一个长期可积累的 **个人知识空间**。
